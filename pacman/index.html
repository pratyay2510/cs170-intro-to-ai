Pac-Man — eat all dots (no ghosts)

Same search ideas: BFS and UCS minimize number of steps; DFS finds some valid path. Goal: every dot eaten. Each move (N/S/E/W) costs 1.

(a) State space: (x, y) for Pac-Man and a boolean per dot (encoded as a bitmask).   (b) Actions: N, S, E, W.   (c) Transition: update position; if a dot is there and still “on,” clear it.   (d) Goal test: all dots cleared.   (e) Path cost: number of steps.
Search code (generic): The next script defines PacManSearch — pure BFS / DFS / UCS that take any parsed grid (walls, dotIndex, dimensions, start cell, dot count). The UI script only parses the maze, then calls PacManSearch.bfs (or dfs / ucs) when you press the run buttons. A solvable maze will return a move list; an unsolvable layout returns no solution.

Build your maze

Set width and height, then choose a tool and click cells. Outer border starts as walls; the interior is empty paths. Place exactly one Pac-Man start (P), at least one dot, and connect paths with Floor so Pac-Man can walk. Maximum 20 dots for search in the browser.
